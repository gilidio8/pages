Equipe e escopo operacional

Projeto pequeno, conduzido por 1 desenvolvedor. A execucao deve ser sequenciada para evitar troca constante de contexto:

Frente Foco Entregaveis
1. Loja e cliente Gestao e loja Conta Play Console, textos, screenshots, politica de privacidade, respostas de declaracoes e comunicacao com o cliente.
2. Aplicativo Android Aplicativo Android Capacitor, appId final, build web, pasta Android, assinatura, AAB, testes em aparelho real e envio da release.
3. Banco e backend Banco de dados e backend Supabase, migracoes, RLS, Edge Functions, variaveis de ambiente, usuarios de teste, backup e validacao de dados.
Para manter o projeto pequeno e viavel para 1 dev, novas funcionalidades devem ser congeladas durante a fase de publicacao. A prioridade passa a ser estabilidade, privacidade, build assinado, banco revisado e revisao da Google.

Configuracao recomendada do Capacitor

Arquivo: capacitor.config.ts.

import type { CapacitorConfig } from '@capacitor/cli';

const config: CapacitorConfig = {
  appId: 'br.com.cliente.vaidemoto',
  appName: 'VAI DE MOTO',
  webDir: 'dist',
};

export default config;
  • Defina o appId antes de publicar pela primeira vez. O package name e permanente na Play Store.
  • Remova server.url para evitar dependencia do Lovable em producao.
  • Remova cleartext: true, salvo necessidade tecnica muito bem justificada.

Ambiente necessario

  • Node.js LTS instalado.
  • Android Studio instalado.
  • Android SDK, Platform Tools e Build Tools instalados pelo SDK Manager.
  • JDK compativel com o Android Gradle Plugin gerado pelo Capacitor.
  • Acesso a conta Play Console do cliente com permissao para releases.
  • Acesso ao projeto Supabase correto, com permissao para banco, Auth, Storage, Edge Functions e variaveis.

Banco de dados Supabase

O projeto usa Supabase no frontend por src/integrations/supabase/client.ts e possui migracoes em supabase/migrations.

Analise: para o tamanho do produto, o banco tem complexidade media. Ele nao e apenas cadastro e login; tambem cobre corrida em tempo real, financeiro, notificacoes, storage, email e regras de acesso por perfil. Antes da Play Store, a revisao do banco deve ser tratada como etapa obrigatoria, nao como detalhe final.
Area O que existe no projeto Validacao antes da Play Store
Autenticacao Usuarios em auth.users, perfis em profiles e papeis em user_roles. Criar usuario de teste para Google e confirmar cadastro, login, recuperacao de senha, papel de passageiro, motorista e administrador.
Operacao rides, driver_details, ride_ratings e status de corrida. Testar passageiro, motorista, aceite de corrida, cancelamento, conclusao, avaliacao, localizacao e historico.
Financeiro platform_settings, driver_transactions e campos de forma de pagamento. Validar valores, comissoes, relatorios e se dados financeiros entram na declaracao de privacidade.
Notificacoes push_subscriptions e funcoes relacionadas a notificacao/email. Confirmar permissao de notificacao no Android, comportamento quando o usuario nega a permissao e duplicidade das migrations de push.
Storage Buckets como avatars e email-assets. Revisar politicas de leitura/escrita e evitar exposicao indevida de documentos ou imagens.

Variaveis obrigatorias

O build precisa receber as variaveis publicas do Supabase:

VITE_SUPABASE_URL=...
VITE_SUPABASE_PUBLISHABLE_KEY=...
  • Nao subir .env para o Git.
  • Configurar variaveis no ambiente que fizer o build.
  • Separar projeto Supabase de teste e producao quando houver dados reais.

Migracoes e seguranca

  • Aplicar as migrations em ordem no ambiente de producao.
  • Conferir se RLS esta ativo nas tabelas que guardam dados de usuario.
  • Validar politicas para passageiro, motorista e administrador.
  • Executar teste com usuarios reais de cada papel antes de enviar para revisao.
  • Confirmar rotina de backup ou exportacao do banco antes de abrir o app para usuarios finais.
  • Revisar se a chave anon/publica esta limitada por RLS e se nenhuma service role key esta exposta no frontend.
  • Confirmar se o banco de producao nao contem dados de teste usados na revisao da Google.

Fluxo de build local

1. Instalar dependencias

npm install

2. Validar build web

npm run build

3. Adicionar Android na primeira vez

npx cap add android

4. Sincronizar em builds seguintes

npm run build
npx cap sync android

5. Abrir no Android Studio

npx cap open android

Target SDK e compatibilidade

O projeto usa Capacitor Android 8.x. Pela matriz oficial do Capacitor, a linha 8.x trabalha com target SDK 36.

Depois de gerar a pasta Android, conferir:

android/variables.gradle

O Google Play exige target API recente para novos apps e atualizacoes. Em 2026, mantenha Capacitor, Android Gradle Plugin e SDK atualizados antes do envio.

Assinatura e chave de upload

  1. Abra o projeto Android no Android Studio.
  2. Acesse Build > Generate Signed Bundle / APK.
  3. Escolha Android App Bundle.
  4. Crie uma keystore se ainda nao existir.
  5. Guarde arquivo .jks, senha da keystore, alias e senha do alias.
  6. Selecione variant release e gere o AAB.

Saida esperada:

android/app/build/outputs/bundle/release/app-release.aab
Nao versionar a keystore no Git. Guarde em cofre de senhas ou armazenamento seguro do cliente. A chave e necessaria para atualizacoes futuras.

Opcao de build via terminal

Depois de configurar assinatura no Gradle, o bundle pode ser gerado assim:

cd android
.\gradlew bundleRelease

Se usar variaveis locais para senhas, mantenha arquivos como keystore.properties fora do Git e valide o .gitignore.

Preparacao da Play Console

  1. Criar app em Home > Create app.
  2. Selecionar idioma padrao, nome, app/jogo, gratis/pago e email de contato.
  3. Aceitar Play App Signing.
  4. Preencher ficha da loja: nome, descricao curta, descricao completa, icone, screenshots e feature graphic.
  5. Preencher categoria, tags e dados de contato.
  6. Adicionar politica de privacidade.
  7. Responder declaracoes de conteudo, seguranca de dados, publico-alvo e permissoes.

Declaracoes que exigem atencao

Declaracao O que verificar
Data safety Dados de usuario, login, localizacao, identificadores, dados de contato, arquivos e compartilhamento com terceiros como Supabase ou Google Maps.
App access Se houver login, fornecer usuario e senha de teste para a revisao da Google.
Permissions Justificar permissoes sensiveis, principalmente localizacao, notificacoes e servicos em primeiro plano.
Content rating Responder questionario de classificacao indicativa de acordo com o conteudo real do app.
Target audience Definir faixa etaria correta e evitar declaracoes incompatíveis com coleta de dados.

Teste antes da producao

  • Rodar o app em aparelho Android real.
  • Testar instalacao a partir da trilha de teste da Play Store, nao apenas via Android Studio.
  • Validar login, cadastro, fluxo principal, mapa/localizacao, notificacoes e telas de erro.
  • Validar leitura e escrita no Supabase usando usuarios de passageiro, motorista e administrador.
  • Testar com internet ruim e sem internet.
  • Confirmar que nenhum badge, ambiente de preview ou URL do Lovable aparece no app final.
Contas pessoais criadas depois de 13/11/2023 podem precisar de teste fechado com pelo menos 12 testadores opt-in por 14 dias continuos antes de solicitar acesso a producao.

Envio do AAB

  1. Ir em Test and release > Testing > Internal testing.
  2. Criar release.
  3. Enviar o arquivo app-release.aab.
  4. Conferir avisos de target SDK, assinatura, permissoes e tamanho.
  5. Adicionar release notes.
  6. Publicar a trilha de teste.
  7. Depois de validar, promover para closed testing ou production.

Atualizacoes futuras

Para cada nova versao enviada a Play Store:

  1. Incrementar versionCode.
  2. Ajustar versionName, se necessario.
  3. Rodar npm run build.
  4. Rodar npx cap sync android.
  5. Gerar novo AAB assinado com a mesma upload key.
  6. Enviar para teste ou producao.
